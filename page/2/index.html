Q&A

Q1: 无法释放锁？

A：加超时时间 expire time ;
Q2: 生成完成后其他请求获取不到锁？

A：要释放锁 delete key;
Q3: 怎么确定你删除的锁是你自己加的锁？

A：给锁增加线程唯一标识，删除前先判断标识是否匹配；
Q4: 超时时间内无法执行完怎么办？

A：通过 watch dog 临近结束前增加超时时间;(Redission)；
Q5: Redis 挂了，由于数据同步异步问题，导致2个线程获得同一把锁怎么办？

A：不用主从，用红锁( redis lock )部署奇数台独立的 Redis，依次获取锁，当成功获取半数以上的锁才视为加锁成功；
Q6: 2个线程都在争锁，其中一台 Redis 挂了，一个请求只要获得4台中的2台就拿到锁了；这时候重启，另一请求拿到了剩下3台的锁，也拿到锁成功？

A：告诉运维，等key都过期了再重启；
Q7: 如果多台 Redis 挂了怎么办？

A：部署多台 Redis 的目的就是为了防止某台挂掉，最后还挂了多台，要把运维开了。

终极解决方案 MySQL 乐观锁 + zk
用 zk 的临时节点当作版本号，存在数据库里。

ZAB协议

  • Zookeeper Atomic Broadcast Zk原子广播

  • 快速领导者选举：谁的数据越新谁就做领导；

    • zxid 相同的时候比较myid；（myid 一开始配置好，配置相同的会报错）
    • zxid 低32位事务 ID 自增，高32位年代 ID epoch；
    • epoch -> zxid -> myid 成为 leader 比较顺序；

  • 节点多了，写效率会下降，所以设计了一种 Observer 状态，只读，不参与领导选举；

2PC

两阶段提交

  1. 接收请求->记录日志（zxid，事物 ID 自增）->持久化
  2. 预提交事务，向 follwer 发送日志；
    1. follower 接到日志并持久化；
    2. 更新 DataTree
  3. 等着接收 follwer 的 ack，判断接受数是否过半；
  4. 提交事务
    1. 向 follower 发送提交事务请求；
    2. 向 Observer 发送提交事务请求；
  5. 更新 DataTree

投票过程

zk1 zk2 observer
zxid = 100 zxid = 100
myid = 1 myid = 2
投票箱 我,投谁,当前 zxid

  1. 机器启动后建立连接，互换选票进行PK比较；第一票都投给自己；
  2. 收到选票后进行比较，把输的票换成赢的信息；
    1. 本来zk1选票是 [1,1,100]，但是zk2比zk1的myid大，所以要选zk2，所以zk1的选票改成了[1,2,100]
    2. 然后再把 [1,2,100] 这张选票发给zk2
