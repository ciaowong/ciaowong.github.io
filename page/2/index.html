索引

偏移量、长度
存储在硬盘中，会预加载进内存；分页中存储着指向下一块索引页的地址；

InnoDB 数据插入的时候跟索引存储在一起；

有主键的话就跟主键存在一起，没有跟唯一键，再没有跟隐藏键 rowid；

聚簇/非聚簇索引 的区别就是看 数据 跟 索引是否存放在一起；

基数 (DV distinct value) 足够多创建才有意义，如果只有0，1，效果很差，反而查询很慢；

  • 索引维护

    如果不是按照自增 id 插入数据，可能会引起页分裂
    比如要一个 id 正好在一个页现有的2个 id 之间，这时候页满了，就要把页拆成2个新的页，把新数据放进去；由下往上，可能上面的索引页也会引起分裂；
    反之删除数据，引起页合并；

  • 联合索引

    一个索引包含多个索引列；

  • 索引覆盖

    根据 name 去 B+ 树上取到了 id,name ，不需要回表，效率高； 这叫索引覆盖；

  • 索引下推 (index_condition_pushdown) 5.7后默认开启：

    没开启:先根据 name 去存储引擎拉去符合结果的数据返回 server 再对 age 进行过滤；
    开启后:直接根据2个条件进行拉取；

    “下推”就是本来是 server 层做的事情推到 存储引擎去做了；

JVM

发表于 2021-04-06 更新于 2021-04-09 分类于 JAVA技术

识别垃圾的方法
1.标记算法，有标记+1，没标记-1，标记为0的时候认为是垃圾
问题：循环依赖，大家都有标记，但是大家都是垃圾，无法处理；
2.根可达算法（RootSearching）
有引用就继续往下找，找不到的都是垃圾

算法
1.标记清楚法(Mark-Sweep)，会导致内存碎片化的问题；
2.Copy算法，会导致浪费空间的问题，每次都只用一半的内存；
3.标记压缩算法，效率比较低

内存布局
|eden(伊甸区)|Survivor0|S1|Tenured|
| young | old |
新生代一次回收的垃圾比较多，要求算法比较快
1.新生代有用的拷到s0，清空新生代；
2.新生代有用的拷到s1，s0拷到s1，清空新生代和s0；
3….
4.存活n次以后的对象移至老年代(ps的age是15，cms的age是6)；

分代垃圾回收器，young old 搭配使用(单线程、多线程)
young old
Serial Serial Old
Parallel Scavenge Parallel Old
ParaNew Concurent Mark Sweep

不分代垃圾回收器
G1, ZGC, Shenandoah

CMS,Concurent Mark Sweep, 并发标记清除，一边工作一边进行垃圾回收；
采用三色标记算法：
1.已经触达且所有字节点都已经触达的标为黑色；
2.已经触达但是有字节点未触达的标为灰色；
3.未触达的标为白色；
存在问题：白色被黑色引用，黑色已经触达不会再扫，所以白色会被回收；
解决：incremental update, 当白色被黑色引用的时候，要重新扫描一遍，重新标记。

JVM Tuning 调试，目的是为了尽量减少 FGC (无法避免)
FGC 导致 STW Stop the world
mirror gc = young gc; major gc = full gc;

Xms Xmx 一致为了避免扩缩容；
