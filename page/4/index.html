Redis

发表于 2021-04-06 更新于 2021-05-12 分类于 JAVA技术

  • 官网 redis.io

  • 最新版本 6.2.1

  • 默认端口 6379

  • NoSQL 非关系型键值对数据库，数据存储在内存

  • C语言实现

  • 5种数据类型，且每种类型都有自己的本地方法，让计算向数据移动

    • string
    • list 双向，实现栈（同向），队列（异向），数组（index）
    • set 去重、无序，实现随机事件、验证码
    • hash 详情页
    • zset 去重、有序，实现排行榜、翻页、动态

  • 单线程，串行化（6.x开始 IO threads 多线程）

  • 快的原因：k v键值对作为缓存，缓存的数据就是不完整的，不要找关联关系。

  • 跟数据库的区别：
    缓存：不重要，可以丢，减轻数据库的压力，到数据库的都是有效请求。
    存储：珍惜。

  • 分布式一致性太难了
    减少你认为的所谓的高效并行

数据类型&结构

hash: ziplist, hashtable; key/value 的长度超过 64 个字节需要将 ziplist 转成 hashtab 对象。

set: hashtable, inset;

zset: ziplist, dict + skiplist;

zk由 Java 编写

zk保证最终一致性

ZAB协议

  • Zookeeper Atomic Broadcast Zk原子广播

  • 快速领导者选举：谁的数据越新谁就做领导；

    • zxid 相同的时候比较myid；（myid 一开始配置好，配置相同的会报错）
    • zxid 低32位事务 ID 自增，高32位年代 ID epoch；
    • epoch -> zxid -> myid 成为 leader 比较顺序；

  • 节点多了，写效率会下降，所以设计了一种 Observer 状态，只读，不参与领导选举；

2PC

两阶段提交

  1. 接收请求->记录日志（zxid，事物 ID 自增）->持久化
  2. 预提交事务，向 follwer 发送日志；
    1. follower 接到日志并持久化；
    2. 更新 DataTree
  3. 等着接收 follwer 的 ack，判断接受数是否过半；
  4. 提交事务
    1. 向 follower 发送提交事务请求；
    2. 向 Observer 发送提交事务请求；
  5. 更新 DataTree

投票过程

zk1 zk2 observer
zxid = 100 zxid = 100
myid = 1 myid = 2
投票箱 我,投谁,当前 zxid
[1,1,100] [2,2,100] 1
[1,2,100],[2,2,100] [2,2,100],[1,2,100] 2
  1. 机器启动后建立连接，互换选票进行PK比较；第一票都投给自己；
  2. 收到选票后进行比较，把输的票换成赢的信息；
    1. 本来zk1选票是 [1,1,100]，但是zk2比zk1的myid大，所以要选zk2，所以zk1的选票改成了[1,2,100]
    2. 然后再把 [1,2,100] 这张选票发给zk2

synchronized

  • 本质是修改了 Object Head 对象头的内容

  • jdk1.5 以前是悲观锁实现的；以后改成了 偏向锁(现在也已经弃用了)-乐观锁-重量级锁

  • 无法保证 有序性

synchronized 和 ReentrantLock

  • 都是可重入锁

  • syn 依赖 jvm，ReentrantLock 是 JDK 实现的

  • ReentrantLock有时间锁等候（tryLock ()），可中断锁等候（lockInterruptibly ()），锁投票等，适合用于高度竞争锁和多个条件变量的地方。

  • 前者提供了可轮询的锁请求，可以尝试去获取锁（tryLock ( )），如果失败，则会释放已经获得的锁。有完善的错误恢复机制，可以避免死锁的发生。

乐观锁 自旋锁 CAS

Compare and Swap

实现思想：1.当前值是1，拿走加1完变成2，回来判断原先值是否是1，如果是就把2写回原处；
2.如果不是，把最新的值x拿走，再加1，回来判断原先值是否是x，如果是就把x+1写回原处，否则重复2；
自旋，如果旋得太久，消耗CPU资源（重量级锁放等待队列，不会消耗CPU资源）

Q1: 会产生 ABA 问题，即回来判断的时候 值已经变过了，又变回来；
解决方案：1.加版本号；2.增加 Boolean 标记判断；

Q2: 怎么保证操作的原子性？
JDK 用了 Unsafe.java compareAndSwap 方法
具体实现是 native Unsafe.cpp C++ 操作 (涉及指针操作，是给开发 JDK 的人用的，所以不安全) Atomic cmpxhg - asm 汇编 底层实现；
通过 LOCK_IF_MP = Lock cmpxchg (cmpxchg 不保证原子性，前面的 Lock 保证)
MP = multi process, 单 CPU 时不加 Lock ，多个才加，以此来保证原子性；

锁的升级过程

new 对象 偏向锁未启动 -> 普通对象 -> 偏向未启动 -> 升级 轻量级锁
偏向锁已启动 -> 匿名对象 -> 超过4s -> 偏向锁 -> 轻度竞争 -> 升级 轻量级锁 -> 重度竞争、耗时 -> 升级 重量级锁 (OS 来操作)

ReentrantLock 一个锁对一个队列，精确唤醒；
CountDownLatch 门闩

不影响单线程最终一致性，可以进行 指令重排（乱序）

线程切换的影响？

内存泄漏、上下文切换、死锁 。

ThreadLocal 原理，是否每个线程都能定义？还是总的就一个？怎么利用 ThreadLocal 制造 OOM？

  • 每一个线程都有自己的专属本地变量；
  • 创建了一个 ThreadLocal 变量，那么访问这个变量的每个线程都会有这个变量的本地副本；
  • threadLocals，inheritableThreadLocals，都是 ThreadLocalMap，是 ThreadLocal 类实现的定制化的 HashMap，变量默认都是 null，调用 get() set() 的时候才创建，最终的变量是放在了当前线程的 ThreadLocalMap 中。
  • ThreadLocalMap 是 ThreadLocal 的静态内部类。
  • ThreadLocalMap 中使用的 key 为 ThreadLocal 的弱引用,而 value 是强引用。ThreadLocal 没有被外部强引用的情况下，在垃圾回收的时候，key 会被清理掉，而 value 不会被清理掉，导致 OOM。用完调用 remove() 释放。
