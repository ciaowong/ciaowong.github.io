    数据库其实也是程序。数据结构+算法

    索引

    偏移量、长度
    存储在硬盘中，会预加载进内存；分页中存储着指向下一块索引页的地址；
    索引维护，如果不是按照自增 id 插入数据，可能会引起页分裂
    比如要一个 id 正好在一个页现有的2个 id 之间，这时候页满了，就要把页拆成2个新的页，把新数据放进去；由下往上，可能上面的索引页也会引起分裂；
    反之删除数据，引起页合并；
    联合索引：一个索引包含多个索引列；
    基数 (DV distinct value) 足够多创建才有意义，如果只有0，1，效果很差，反而查询很慢；

    一页默认大小 16kb 16384

    InnoDB 数据插入的时候跟索引存储在一起；
    有主键的话就跟主键存在一起，没有跟唯一键，再没有跟隐藏键 rowid；

    聚簇/非聚簇索引 的区别就是看数据跟索引是否存放在一起；

    数据结构 B+树

    3-4层的 B+ 树就能支持 千万级别的 数据量
    树要足够矮，减少检索的次数
    保证key尽可能小地占用空间
    叶子节点存储所有插入的数据
    叶子节点之间存在指针，双向的
    叶子节点有多个元素且排好序的

    局部原理

    每次读取，周边一定范围内的数据也会被读到，一起加载进内存；
    innodb 每页默认大小 16kb

    减少IO

    表a 有 id name age id 主键 name 索引
    select * from table where id = ? ; 可以根据主键直接找到记录；

    回表

    ：where name = ?; 1.根据索引找到主键，再根据主键找到记录，这叫回表；（效率不高，IO

    索引覆盖

    ：select id,name from table where name = ?; 根据 name 去 B+ 树上取到了 id,name ，不需要回表，效率高； 这叫索引覆盖；

    最左匹配

    ： where name = ? 符合；
    where name = ? and age = ? 符合；
    where age = ? and name = ? 符合；（mysql 优化器会进行优化，因为调整顺序不影响查询结果）
    where age = ? 不符合；

    索引下推 (index_condition_pushdown)：

    没开启之前，先根据 name 去存储引擎拉去符合结果的数据返回 server 再对 age 进行过滤；
    开启后，直接根据2个条件进行拉取；

    client
    server 连接、分析、优化、执行
    存储引擎 innodb myisam

    下推的意思就是本来是 server 层做的事情推到 存储引擎去做了；5.7后默认开启
