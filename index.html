电商超卖问题

发表于 2021-04-07 分类于 JAVA技术

怎么防止电商超卖？

情况1: 单机，高并发，用 synchronized 关键字加锁；
情况2: 多机，引入中介防止订单重复：
中介1: 生成订单通过 MySQL 主键，防止重复；
中介2: 通过 Redis setnx key value
Q1: 无法释放锁，所以加超时时间 expire time ;
Q2: 生成完成后要释放锁 delete key;
Q3: 怎么确定你删除的锁是你自己加的锁？给锁增加线程唯一标示；
Q4: 超时时间内无法执行完怎么办？通过 watch dog 临近结束前增加超时时间;(Redission)；
Q5: Redis 挂了，由于数据同步异步问题，导致2个线程获得同一把锁怎么办？部署奇数台 Redis，依次获取锁，当成功获取半数以上的锁才视为加锁成功；
Q6: 2个线程都在争锁，其中一台 Redis 挂了，一个请求只要获得4台中的2台就拿到锁了；这时候重启，另一请求拿到了剩下3台的锁，也拿到锁成功？告诉运维，等key都过期了再重启；
Q7: 如果多台 Redis 挂了怎么办？部署多台 Redis 的目的就是为了防止某台挂掉，最后还挂了多台，要把运维开了。

终极解决方案 MySQL 乐观锁 + zk

采用三色标记算法：
1.已经触达且所有字节点都已经触达的标为黑色；
2.已经触达但是有字节点未触达的标为灰色；
3.未触达的标为白色；
存在问题：白色被黑色引用，黑色已经触达不会再扫，所以白色会被回收；
解决：incremental update, 当白色被黑色引用的时候，要重新扫描一遍，重新标记。

多线程

发表于 2021-04-06 更新于 2021-04-12 分类于 JAVA技术

synchronized 关键字，无法保证 有序性

本质是修改了 Object Head 对象头的内容
jdk1.5 以前是悲观锁实现的
以后改成了 偏向锁(现在也已经弃用了)-乐观锁-重量级锁

乐观锁也叫自旋锁 CAS Compare and Swap

实现思想：1.当前值是1，拿走加1完变成2，回来判断原先值是否是1，如果是就把2写回原处；
2.如果不是，把最新的值x拿走，再加1，回来判断原先值是否是x，，如果是就把x+1写回原处，否则重复2；
自旋，如果旋得太久，消耗CPU资源

Q1: 会产生 ABA 问题，即回来判断的时候 值已经变过了，又变回来；
解决方案：1.加版本号；2.增加 Boolean 标记判断；

Q2: 怎么保证操作的原子性？
JDK 用了 Unsafe.java compareAndSwap 方法
具体实现是 native Unsafe.cpp C++ 操作 (涉及指针操作，是给开发 JDK 的人用的，所以不安全) Atomic cmpxhg - asm 汇编 底层实现；
通过 LOCK_IF_MP = Lock cmpxchg (cmpxchg 不保证原子性，前面的 Lock 保证)
MP = multi process, 单 CPU 时不加 Lock ，多个才加，以此来保证原子性；

锁的升级过程

new 对象 偏向锁未启动 -> 普通对象 -> 偏向未启动 -> 升级 轻量级锁
偏向锁已启动 -> 匿名对象 -> 超过4s -> 偏向锁 -> 轻度竞争 -> 升级 轻量级锁 -> 重度竞争、耗时 -> 升级 重量级锁 (OS 来操作)
