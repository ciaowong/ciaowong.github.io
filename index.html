Q1: ConcurrentHashMap 底层结构？

保证线程安全的方案：

  • JDK1.8：synchronized+CAS+HashEntry + 红黑树；

  • JDK1.7：ReentrantLock+Segment+HashEntry。

volatile 修饰了 HashEntry ，保证了多线程环境下数据获取时的可见性！

key 和 value 是都不能为 null 的。在并发的情况下带来难以容忍的二义性。

  1. 这个 key 从来没有在 map 中映射过。
  2. 这个 key 的 value 在设置的时候，就是 null。

1.7: ConcurrnetHashMap 由很多个 Segment 组合，egment 的个数一旦初始化就不能改变，默认 Segment 的个数是 16 个。

Q2: CMS 回收过程？

标记**-**清除，边工作边垃圾回收。

三色标记法；初始标记->并发标记->重新标记->并发清除，1，3会发生 STW；

Q3: 讲讲 G1 垃圾回收器？

面向服务器的垃圾收集器，主要针对配备多颗处理器及大容量内存的机器。降低 GC 停顿时间，还具备高吞吐量性能特征。

初始标记->并发标记->最终标记->筛选回收；

在后台维护了一个优先列表，每次根据允许的收集时间，优先选择回收价值最大的区 (Garbage First)。

盒伙人项目总结

发表于 2021-04-28 更新于 2021-04-29 分类于 项目

从0到1主导了盒伙人业绩统计的优化。

场景

处理数据量级

Before: 一天处理20w 数据，生成3份统计报表；

After：一天处理200w+ 数据，生成 30+ 份报表。

系统架构

Before: 旧系统单机，处理逻辑简陋、效率低下、逻辑冗余；

After: 采用分布式系统，准实时统计，实现三高。

生产事故

Before: 每个月都会出几次问题；

After: 一年来0事故。

改进事项

源数据不准确

Before:因为源头清分签约价数据不准确，且旧的逻辑更新签约价的方式是将记录全部删除后重新生成；

After:将交易额与签约价进行解耦，独立操作，互不影响。

统计逻辑冗余

Before:日报从源数据生成，月报也从源数据生成，产生了大量重复性的计算；

After:源数据生成日报，月报由日报进行汇总。

Before:每次计算团队报表需要从DB获取完整的层级关系；

After:将层级关系放入 Redis 缓存。

可用性

Before:旧系统1天只进行1次统计运算，一锤子买卖，错了就要人工干预；

After:改进后每小时进行统计，并检查统计结果，做了补偿机制，结合企业微信机器人进行通知，对生产数据做到早发现早治疗。

遇到的问题与解决

分布式

Q1:在演进过程中，为了在分布式环境下实现删除逻辑，采用了分布式锁；

A1:起初用最简单的 setNx 实现，遇到了超时时间内无法删除完成删除数据，释放锁后下一个任务生成的数据又被删除，导致最终数据缺失；后改用了 Redisson 框架实现分布式锁；最终还是通过优化业务逻辑规避了删除动作；

A2:为了加快统计运算，采用了 ElasticJob 进行分布式运算，通过 ID 取模的方式将记录进行多线程处理；

Q2:怎么知道系统统计数据正确，运行正常？

A1:通过定时任务跑 SQL 检查数据库表数据是否跑平，通过企业微信机器人进行通知。

DB

Q1:分页插件没配置好，每次都会进行 count(1) 操作；分页语句没有绑定变量，Oracle 执行慢；

A1:去除分页插件，手动实现分页查询；

Q2:慢查询、SQL不走索引；

A1:数据表按日期进行分区，相当于磁盘路径，快；

A2:查询条件使用了函数，走不了索引，将函数放到等号的右边（拓展索引、B+树相关）

A3:分页条数设置过小，还停留在10、20条web管理页面的时代；分页不会提高查询速度，页数越小，查询次数越多，速度越慢。

A4:应用与DB不在一个机房，跨机房有延迟，虽然平时感知不明显，但是操作次数多了以后延迟很明显；所以将应用迁移至同一机房；

JVM

Q1:8G内存的服务器给应用实际才分配了2G内存；

A1:配置从测试环境 copy ，没有检查；（拓展JVM相关）

上下游部门配合事项

Q1:生产相关信息同步不及时，导致业绩统计不准；

A1:将相关人员拉到一个企业微信群，群内及时同步影响业绩统计的重要信息，串联上下游。

Redis和db数据怎么保持一致？

方式1：数据库保存数据，redis不persist
redis启动后，从数据库加载数据
不要求强一致实时性的读请求，都由redis处理
要求强一致实时性的读请求，由数据库处理
写请求有2种处理方式，由数据库处理

  • 应用先写到数据库，然后更新redis
  • 应用先写到数据库，然后其它daemon同步到redis
    优点：redis启动不用处理redis数据和数据库不一致
    缺点：redis启动给数据库很大的读压力

方式2：数据库和redis分别处理不同的数据类型
数据库处理要求强一致实时性的数据，例如金融数据、交易数据

Redis处理不要求强一致实时性的数据，例如网站最热贴排行榜

Redis和MySQL数据的同步，代码级别大致可以这样做：
读: 读redis->没有，读MySQL->把mysql数据写回redis
写: 写mysql->成功，写redis

并发不高的情况：
读: 读redis->没有，读mysql->把mysql数据写回redis，有的话直接从redis中取；
写: 写mysql->成功，再写redis；

并发高的情况：
读: 读redis->没有，读mysql->把mysql数据写回redis，有的话直接从redis中取；
写：异步话，先写入redis的缓存，就直接返回；定期或特定动作将数据保存到mysql，可以做到多次更新，一次保存；

目前项目中 API 包内一般都会有 dubbo 的配置文件，如果在引用方的 AppConfig 的 classpath 加了 * 号, 代表加载的时候会扫描 jar 包里面的文件。那么自己就可以不写 dubbo 声明文件，反之得加。建议不扫描jar包,一来扫描jar包可能会引用很多依赖有很多问题，二来自己写一下声明文件也比较清楚引用了哪些东西。
